Research Synthesis → Design Brief
D01 · v1.0
Synthesize raw UXR notes, interview transcripts, or workshop outputs into a structured design brief. Used at the end of Discovery to transition cleanly into the Design phase in Figma.
Discovery Designer Design Brief All Platforms
# Prompt D01 — Research Synthesis → Design Brief # Tool: Claude.ai · Phase: Discovery → Design ## Inputs RESEARCH_INPUT: [paste interview notes, survey results, workshop sticky notes, heatmap findings, analytics highlights — any format] PROJECT_NAME: [client / project] SCOPE: [e.g. "Homepage redesign", "Checkout flow", "Account dashboard"] PLATFORM: [Sitecore / Salesforce / Shopify / Headless] BUSINESS_GOALS: [conversion targets, engagement goals, client KPIs] ## Task Synthesize RESEARCH_INPUT into a structured design brief for SCOPE. 1. User Insights (top 5–7) Each insight: one declarative sentence + 2–3 supporting evidence quotes or data points. Format: insight → evidence → design implication. Distinguish between: observed behavior vs. stated preference vs. inferred need. 2. Jobs To Be Done Primary JTBD: "When _____, I want to _____, so I can _____." Secondary JTBDs: (2–3 supporting jobs) Anxieties and friction points observed in research. 3. Design Principles (3–5) Derived directly from insights — not generic best practices. Each principle: name + one-sentence rationale + one design implication. Example format: "Reduce before reveal — users felt overwhelmed by options early. Implication: progressive disclosure on complex flows." 4. Scope Definition In scope: [list] Out of scope: [list — critical to define] Open questions needing design exploration: [list] 5. Success Criteria How will we know the design worked? Tie to BUSINESS_GOALS with specific measurable indicators. 6. Figma Starting Point Recommended first Figma frames to create based on the research. Priority components to design first (highest user impact, highest business value). ## Output Save to: /briefs/PROJECT_NAME-design-brief.md Share with client for alignment before Figma work begins.
Input: raw research · Output: brief shared before Figma starts
Competitive & Analogous Design Pattern Audit
D02 · v1.0
Analyze competitor sites or analogous experiences to extract design patterns, interaction conventions, and strategic opportunities. Feeds directly into Figma exploration.
Discovery Designer Pattern Audit All Platforms
# Prompt D02 — Competitive + Analogous Design Audit # Tool: Claude.ai (with web search) · Phase: Discovery ## Inputs FOCUS_AREA: [e.g. "checkout flow", "product filtering", "onboarding", "navigation"] COMPETITORS: [list URLs or names — 3–5 direct competitors] ANALOGOUS: [list URLs of non-competitor but relevant experiences — e.g. best-in-class examples from other industries: "Airbnb search", "Linear onboarding", "Stripe dashboard"] OUR_CONTEXT: [brief description of our product, user, and constraint] ## Task Analyze FOCUS_AREA across competitors and analogous references. 1. Pattern Inventory For each site/experience: - What pattern do they use for FOCUS_AREA? - What interaction model? (scroll, click, filter, wizard, etc.) - What does it do well? What friction exists? Rate: Strong reference / Partial reference / Anti-pattern to avoid. 2. Convention Map What patterns appear across 3+ of the references? → Established convention. What patterns appear in only 1–2? → Differentiator opportunity. What pattern is conspicuously absent? → Unmet standard or innovation space. 3. Interaction Principles Extracted 5–7 design principles derived from what works across the references. Format: principle name + what the evidence shows + design implication for us. 4. Figma Exploration Recommendations Based on this audit, what 2–3 distinct design directions should be explored? For each direction: name, one-line description, key interaction hypothesis. These become named explorations in the Figma file. 5. Risks & Watch-Outs Patterns that look attractive but have known UX pitfalls. Patterns that are technically complex relative to OUR_CONTEXT. ## Output Save to: /briefs/FOCUS_AREA-competitive-audit.md Pin to Figma cover frame as reference before exploration begins.
Uses web search · Run before Figma exploration begins

Figma Component Auto-Documentation
D03 · v1.1
Read a Figma component and generate complete developer-facing documentation — usage guidelines, variant inventory, do/don't examples, and accessibility notes. Eliminates manual annotation work.
Design Designer Figma MCP Component Docs
# Prompt D03 — Figma Component Auto-Documentation # Tool: Claude Code + Figma MCP · Phase: Design ## Inputs FIGMA_URL: [Figma component URL — master component, not an instance] COMPONENT_NAME: [e.g. "Button", "Card", "Navigation", "Form Field"] DS_NAME: [design system name, if part of one] AUDIENCE: [Developers / Designers / Both] ## Task Using the Figma MCP, read every variant, property, and layer of COMPONENT_NAME at FIGMA_URL. Generate complete component documentation. 1. Component Overview Purpose: what does it do and when should it be used? Component type: atomic / molecule / organism / template Figma path: [component set path in the DS] 2. Variant Inventory Table: Variant Property | Values | Visual Description | When to Use List every variant combination with a plain-English description of what changes. Flag any variant combinations that are invalid or discouraged. 3. Properties / API For developer handoff — list every configurable property: Property Name | Type | Default | Description Mark properties that affect layout (size/spacing changes) vs. only visual. 4. Usage Guidelines Do: [3–5 correct usage examples with context] Don't: [3–5 anti-patterns with explanation of why] Pairing rules: [what other components it works with / doesn't work with] 5. Spacing & Sizing Internal padding, gap values, min/max width/height. How it behaves in a flex/grid container. Truncation rules for text properties. 6. Accessibility Notes ARIA role, required attributes, keyboard behavior. Contrast ratio for each state (reference the color tokens used). 7. Content Guidelines Character limits for each text property. Tone guidance if applicable. ## Output Save to: /design-system/docs/COMPONENT_NAME.md Can be added as a Figma description or published to Storybook.
Uses: Figma Console MCP · Run after component is finalized
Design System Coverage Audit
D04 · v1.0
Audit a Figma design file against the design system to find coverage gaps, detached components, hardcoded values, and token drift. Run before handing designs to developers.
Design Designer Figma MCP Audit Report When DS exists
# Prompt D04 — Design System Coverage Audit # Tool: Claude Code + Figma Console MCP · Phase: Design → Spec ## Inputs DESIGN_FILE_URL: [Figma file or page URL to audit] DS_LIBRARY_URL: [Figma library file URL — the DS source of truth] SCOPE: [e.g. "All frames on page 3 — Homepage designs"] PLATFORM: [Sitecore / Salesforce / Shopify — for token mapping context] ## Task Using the Figma MCP, compare DESIGN_FILE_URL against DS_LIBRARY_URL. Identify every place where the design deviates from or is not covered by the DS. 1. Detached Components List every instance that was once a DS component but is now detached. Per instance: frame name, component name, last known version. Severity: High (core component) / Medium (molecule) / Low (utility). 2. Hardcoded Values Colors not using DS color tokens: list value, location, nearest DS token. Typography not using DS text styles: list font/size/weight, location, nearest DS style. Spacing values not on the DS scale: list px value, location. 3. Missing DS Components UI patterns in the design that the DS doesn't have components for. For each: name, description, how many times it appears. Recommendation: add to DS / build one-off / use existing component differently. 4. Token Drift Any design tokens that are being used inconsistently or out-of-spec. E.g. using primary-500 where semantic-action should be used. 5. Remediation Priority List Ranked list of fixes: most impactful to least. Quick fixes (swap to DS component): [list] Design decisions needed (DS gap): [list] Document as-is and accept (justified deviation): [list] 6. Pre-Handoff Checklist [ ] Zero detached High-severity components [ ] All colors using DS tokens [ ] All type styles using DS text styles [ ] Missing DS components flagged for DS team ## Output Save to: /design-reviews/SCOPE-ds-audit-[date].md Block spec generation until High-severity detachments are resolved.
Run before P01 · Prevents token drift in generated specs
Interaction & Motion Spec for Developer Handoff
D05 · v1.0
Document micro-interactions, transitions, and animation patterns from a Figma prototype into developer-ready specs. Turns prototype motion into implementable CSS/JS instructions with timing, easing, and trigger details.
Design Designer Figma MCP Motion Spec
# Prompt D05 — Interaction & Motion Spec for Developer Handoff # Tool: Claude Code + Figma MCP · Phase: Design → Development ## Inputs PROTOTYPE_URL: [Figma prototype URL with interactions defined] COMPONENT_SCOPE: [which frames/components to document — e.g. "Navigation, Modal, Accordion"] PLATFORM: [Sitecore / Salesforce / Shopify — affects CSS/JS approach] MOTION_LIBRARY: [CSS only / Framer Motion / GSAP / None — project's animation stack] ## Task Using the Figma MCP, read all interactions and transitions in COMPONENT_SCOPE. Translate each into developer-implementable specifications. For each interaction, document: Trigger: [click / hover / focus / scroll / load / swipe] Target element: [element name from Figma layer] Animation type: [fade / slide / scale / rotate / color change / layout shift] Direction: [in / out / from top / from right / expand / collapse] Duration: [ms] — convert Figma's animation duration to milliseconds Easing: [ease-in / ease-out / ease-in-out / spring / custom cubic-bezier] Delay: [ms — if staggered] Trigger condition: [what state must be true for this to fire] Reverse: [does it animate back on trigger release?] CSS / JS implementation hint: Translate to concrete implementation in MOTION_LIBRARY: - CSS only: provide transition / animation / @keyframes block - Framer Motion: provide animate / variants / transition props - GSAP: provide gsap.to() call with full parameters State Machine For complex components (nav, modal, accordion, carousel): Draw the state machine: idle → hover → active → loading → complete → error Document what animation fires on each state transition. Reduced Motion For every animation defined: what is the @media (prefers-reduced-motion) fallback? Never leave this blank. Minimum: instant transition with no easing. ## Output Save to: /specs/interactions/COMPONENT_SCOPE-motion-spec.md Attach to Figma as a component description or dev mode annotation.
Uses: Figma MCP · Feeds into xcm-component-gen (P13/P14)

Rapid HTML Prototype from Design Brief
D06 · v1.1
Generate a working HTML/CSS/JS prototype directly from a design brief or Figma exploration. Used to validate UX hypotheses with stakeholders before investing in full Figma comp work.
Prototyping Designer Claude Code HTML Prototype
# Prompt D06 — Rapid HTML Prototype from Design Brief # Tool: Claude Code (or Claude.ai Artifact) · Phase: Prototyping ## Inputs BRIEF: [paste design brief excerpt, or describe the UX hypothesis to test] FIGMA_URL: [optional — reference frames if Figma explorations exist] INTERACTION_FOCUS: [what specific interaction to validate — e.g. "filtering pattern", "onboarding step flow", "search experience", "cart interaction"] FIDELITY: [Low (structure only) / Mid (realistic content + basic styles) / High (near-production)] DS_TOKENS: [optional — paste color/type tokens to keep it on-brand] ## Task Build a working HTML prototype focused on INTERACTION_FOCUS. This is for stakeholder validation — prioritize the interaction being tested over pixel perfection. Structure Single HTML file. Vanilla JS only — no frameworks. Realistic placeholder content (not "Lorem ipsum" — write plausible real content). Do NOT reproduce Figma designs exactly. Build the interaction hypothesis. Fidelity guidance Low: wireframe-quality, grayscale, interaction logic only. Focus: does the flow make sense? Mid: realistic colors from DS_TOKENS or sensible defaults, real-looking content. Focus: does this feel right in context? High: production-quality styling, responsive, motion design. Focus: is this ready for client presentation? Interaction requirements Every interactive element must actually work — no dead links, no placeholder buttons. Include at least one error or edge case state. If testing a multi-step flow, all steps must be navigable. Prototype annotation Include a collapsible panel or tooltip system that shows: - What hypothesis is being tested at each interaction point - What a "pass" looks like (user achieves goal) - What a "fail" looks like (user is confused or blocked) Cursor usage If building in Cursor: save to /prototypes/INTERACTION_FOCUS/index.html. Cursor can iterate on this file with "refine the filter interaction" follow-up prompts. ## Output Single HTML file: /prototypes/INTERACTION_FOCUS-prototype.html Include README section at top of file: hypothesis, test instructions, success criteria.
Build in Claude.ai Artifacts or Cursor · Iterate before full Figma comp
AI Design Critique & Pre-Handoff Review
D07 · v1.2
Run an AI-assisted design review before sending designs to spec. Evaluates against the design brief, UX principles, accessibility, and platform constraints. Catches issues before they become spec errors.
Design → Spec Designer Figma MCP Design Review
# Prompt D07 — AI Design Critique & Pre-Handoff Review # Tool: Claude Code + Figma MCP · Phase: Design → Spec & Generation ## Inputs FIGMA_URL: [Figma frame(s) to review — link all breakpoints if responsive] DESIGN_BRIEF: /briefs/PROJECT_NAME-design-brief.md [from D01] DESIGN_PRINCIPLES: [paste 3–5 principles from the brief, or list key constraints] PLATFORM: [platform — for platform-specific feasibility checks] REVIEW_TYPE: [Full critique / UX only / Accessibility only / Platform feasibility only] ## Task Using the Figma MCP, review FIGMA_URL. Be a critical and honest reviewer. The goal is to catch problems before they propagate into specs. 1. Brief Alignment Does each design decision in the Figma serve the jobs-to-be-done in DESIGN_BRIEF? List any design elements that seem misaligned with the brief. What design decisions are not explained by the brief? 2. UX Clarity & Flow Primary action hierarchy: is the most important action the most visually prominent? Information hierarchy: does the eye move in the right order? Wayfinding: can a user tell where they are and what to do next? Error prevention: are there states that will confuse users? Flag each issue: [UX ISSUE: description] | Severity: Critical / Major / Minor 3. Accessibility Pre-check (Applies accessibility-rules.md constraints) Heading structure visible in the design. Touch target sizes (minimum 44×44px on mobile). Color contrast — flag any text/background combinations that look close to the 4.5:1 threshold. States: are focus and hover states visible in the design? Any interactive elements that will require complex ARIA? 4. Platform Feasibility Given PLATFORM, flag anything that will be technically complex or impossible: - Patterns that the platform's rendering model doesn't support natively - Interactions that require custom development (not available in standard components) - Content model implications: anything that looks editable but would require unusual CMS setup 5. Missing States What states are not designed? (Loading, error, empty, max content, long text) These will become [DESIGN GAP] flags in the spec — better to design them now. 6. Verdict Ready for spec: Yes / No / Yes with caveats Blocking issues (must resolve before spec): [list] Non-blocking notes (address in this sprint): [list] Deferred items (backlog): [list] ## Output Save to: /design-reviews/FIGMA_URL-critique-[date].md Share with designer for resolution, then proceed to P01.
Run before P01 · Reduces L1 validation failures downstream
Responsive Layout Exploration in Cursor
D08 · v1.0
Use Cursor + Claude to rapidly generate and compare responsive layout variations for a component or page section. Faster than exploring in Figma for layout-heavy decisions. Outputs go back into Figma as reference.
Prototyping Designer Cursor Layout Variants
# Prompt D08 — Responsive Layout Exploration in Cursor # Tool: Cursor Agent · Phase: Prototyping → Design ## Inputs COMPONENT: [what you're exploring — e.g. "Card grid", "Hero section", "Filter + results layout"] CONTENT_MODEL: [what content needs to fit — list the actual content elements and rough volumes] CONSTRAINTS: [design constraints — e.g. "must work without JS", "Tailwind only", "max 3 columns", "sticky sidebar on desktop"] DIRECTION_COUNT: [how many layout directions to generate — recommend 3] DS_TOKENS: [paste color/spacing tokens for visual consistency] ## Task In Cursor, generate DIRECTION_COUNT distinct layout approaches for COMPONENT. Each must genuinely solve the layout problem differently — not just minor variations. For each layout direction: File: /prototypes/layouts/COMPONENT-direction-[N].html Name the approach (e.g. "Sidebar fixed", "Cards masonry", "Stacked progressive"): - Desktop (1440px): full layout - Tablet (768px): how it adapts — be specific about what moves - Mobile (375px): final mobile layout — not just "stack everything" Rules: - Tailwind CSS only (or paste specific DS_TOKENS) - Realistic content — use actual words from CONTENT_MODEL - Show at least one edge case: long title, missing image, empty state Comparison page: File: /prototypes/layouts/COMPONENT-comparison.html Side-by-side view of all DIRECTION_COUNT approaches at each breakpoint. Add decision labels: What problem does each approach solve best? Cursor iteration loop: After generating, iterate with: "Direction 2 feels right but the mobile stacking is wrong — make the title come first, then the image." "Combine the sidebar approach from Direction 1 with the card style from Direction 3." Back to Figma: Once a direction is chosen, use D03 (Figma Component Auto-Doc) to capture the chosen approach as Figma component specs. Screenshot the chosen prototype frame as a Figma cover reference. ## Output /prototypes/layouts/COMPONENT-direction-[1/2/3].html /prototypes/layouts/COMPONENT-comparison.html Decision captured in: /briefs/PROJECT_NAME-layout-decisions.md
Run in Cursor Agent mode · Iterate before committing to Figma

## Salesforce Conventions ### Governor Limit Rules (Hard Limits) - ZERO SOQL queries inside for-loops — ever. This is a blocker in every code review. - All DML operations inside try/catch with meaningful error messages. - @AuraEnabled(cacheable=true) for all read-only Apex methods. - WITH SECURITY_ENFORCED on all SOQL for FLS enforcement. - Maximum 100 DML statements per transaction — design for bulk. ### LWC Conventions - One component per folder: .html / .js / .css / .js-meta.xml - Custom events: kebab-case names (e.g. fire-custom-event, not fireCustomEvent). - @track no longer needed for object/array reactivity (since API v39) — remove it. - No direct DOM manipulation — use reactive properties and @track. - Use @wire before writing imperative Apex. Prefer standard adapters. ### Apex Conventions - PascalCase class names. camelCase methods. - Every class has a paired [ClassName]Test.cls with @isTest annotation. - @TestSetup for shared test data. - Test coverage minimum: 75% (required), target 90%+. - No hardcoded IDs, record types, or strings. Use Custom Metadata or Custom Labels. ### Experience Cloud - Guest user access: explicitly configure OWD + sharing rules. ## Common Errors — Learned Negative Examples ### Spec Generation Errors - DO NOT add fields to a spec that aren't visible in the Figma design. (Common: "Sort by" dropdown, "Save for later", "Compare" buttons not in design) - DO NOT write "responsive" without specifying what changes at each breakpoint. - DO NOT document only the happy path. Always include error, empty, and loading states. - DO NOT assume interactions Figma doesn't show. Flag as [DESIGN GAP] instead. - DO NOT use platform field types from memory — verify against constraint block every time. ### Sitecore-Specific Errors - DO NOT use the Memo field type (not available in XM Cloud). - DO NOT access field.value directly (breaks Experience Editor). - DO NOT generate YAML with duplicate GUIDs. - DO NOT skip registering the component in componentFactory. ### Salesforce-Specific Errors - DO NOT write SOQL inside for-loops — ever. - DO NOT invent object or field API names. Mark as [CONFIRM] and stop. - DO NOT create an Apex class without a paired test class. - DO NOT use @track for primitive reactive properties (deprecated pattern). ### User Stories Errors - DO NOT write acceptance criteria without specifying the breakpoint. Wrong: "When I click the button, the modal opens." Right: "Given I am on desktop, when I click the button, the modal opens." - DO NOT write implementation details in AC. Behavior only. - DO NOT omit the error and empty state stories — offshore needs them. ### Handoff Errors - DO NOT hand off a spec with unresolved [CONFIRM] items. - DO NOT hand off without running L1 validation first. - DO NOT assume offshore has environment access — confirm explicitly.
